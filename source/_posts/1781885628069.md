> 目标市场：日本东京首都圈，第一阶段聚焦在日华人用户与生活应急服务  

---

## 0. 一句话总结

一个面向日本本地生活场景的即时上门服务平台，帮助用户在生病、加班、出差、宠物照顾、临时家务、代办跑腿等场景下，快速找到经过认证、可追踪、可评价、可管理的服务人员上门解决问题。

项目第一阶段不追求“万能平台”，而是先以东京华人市场为切入口，围绕“生活应急代办 + 宠物基础照顾 + 临时家务整理”三类低许可风险、高频刚需服务，完成市场验证。

---

## 1. 项目背景

### 1.1 灵感来源

最初场景来自一个非常真实的生活问题：

```text
对象生病了，想立刻给她买粥、买药、送到家里。
但是临时发现：
- 外卖平台没有卖粥的；
- 想不到买了药谁能给送；
- 朋友不一定方便；
- 想了一圈之后，发现这个问题并没有任何可靠的解决办法。
```

最终发现，这个问题背后并不是“买粥”本身，而是：

```text
人在某个生活瞬间，突然需要另一个可信赖的人，帮自己完成一件具体的小事。
```

由此可以扩展出更多场景：

| 场景 | 用户问题 | 可提供服务 |
|---|---|---|
| 生病 | 不方便出门 | 买粥、买药、买生活用品 |
| 加班 | 没时间处理生活事务 | 代取快递、代购物、送文件 |
| 出差/回国 | 家里无人照看 | 浇花、喂猫、铲屎、代收邮件 |
| 宠物 | 工作忙无法照顾 | 遛狗、喂食、清洁宠物用品 |
| 搬家/整理 | 临时缺人手 | 简单搬运、收纳整理、垃圾处理 |
| 独居/高龄 | 行动不便 | 买菜、代办、陪同前往公共窗口 |
| 留学生/旅游的人 | 初到日本 | 接机、带路、陪同前往公共窗口 |

## 2. 项目定位

### 2.1 不做什么

第一阶段不要把项目定位成以下几类：

```text
不是单纯家政公司；
不是外卖平台；
不是搬家公司；
不是介护公司；
不是出租车/接送服务；
不是医疗服务平台。
```

这些方向要么竞争激烈，要么许可证复杂，要么安全责任极高。

### 2.2 要做什么

第一阶段定位为：

> 面向日本生活场景的“即时生活协助平台”。

核心价值：

```text
用户端：有事时，能找到可信赖的人帮忙。
服务端：有空时，能通过合规方式获得收入。
平台端：用流程、认证、评价、保险、录像授权与客服介入，降低陌生人上门服务的信任风险。
```

### 2.3 项目愿景

短期：

```text
在东京华人圈完成服务需求验证，建立第一批可信服务人员与取得信任的用户。
```

中期：

```text
从东京华人市场扩展到东京本地多语言用户，形成标准化服务平台。
```

长期：

```text
沉淀生活服务流程、服务数据、质量标准和家庭场景数据，未来连接AI管家、远程服务、家庭机器人等生态。
```

## 3. 目标用户分析

### 3.1 需求方用户

#### 3.1.1 在日华人工作者

特点：

- 工作时间长；
- 日语沟通成本高；
- 生活圈相对集中；
- 愿意为节省时间和降低麻烦付费；
- 对中文服务有天然信任和便利需求。

典型需求：

```text
买药、买饭、代取快递、搬家帮手、宠物照顾、手续代办、临时家务。
```

#### 3.1.2 留学生与刚来日本的人

特点：

- 熟悉日本生活流程需要时间；
- 遇到突发问题时缺少资源；
- 对价格敏感，但更需要便利和中文帮助。

典型需求：

```text
搬家、家具组装、代办手续、陪同咨询、生活用品采购。
```

#### 3.1.3 独居人群

特点：

- 生病时没人照顾；
- 出差时宠物和家中事务无人处理；
- 临时需要帮助时缺少稳定支持网络。

典型需求：

```text
临时买饭买药、宠物照顾、代收快递、紧急跑腿。
```

#### 3.1.4 家庭用户

特点：

- 家务负担重；
- 孩子、宠物、老人事务多；
- 对服务人员安全性要求极高。

典型需求：

```text
简单家务、整理收纳、代购物、宠物服务。
```

> 注意：儿童看护、老人介护、医疗护理第一阶段不开放，只作为未来持证业务方向研究。

## 4. 市场切入策略

### 4.1 为什么先做东京华人市场

理由：

1. 用户痛点明确：语言、时间、生活支持不足；
2. 获客路径明确：微信群、小红书、LINE群、朋友介绍；
3. 服务人员供给存在：留学生、自由职业者、主妇、兼职者、周末副业者；
4. 创始人资源匹配：熟悉日本IT、东京生活、华人工作圈。

### 4.2 初始区域建议

选择：

- 华人居住相对集中地区；
- 房租与生活圈更适合初期服务；
- 区域之间距离可控；
- 可以进行30-60分钟内的应急服务。

### 4.3 第一阶段核心策略

```text
先做小区域、高信任、强运营、人工调度。
不要一开始追求全国化、全品类、全自动化。
```

## 5. 服务范围设计

### 5.1 第一阶段开放服务

#### A. 生活应急代办

| 服务 | 示例 | 风险等级 | 是否开放 |
|---|---|---:|---|
| 买饭送饭 | 买粥、买便当、送到指定地点 | 低 | 开放 |
| 代购物 | 便利店、超市、生活用品 | 低 | 开放 |
| 代取快递 | 宅配BOX、指定柜台、便利店取件 | 中 | 开放，但需身份确认 |
| 送文件/送物品 | 公司文件、钥匙、小件物品 | 中 | 开放，但需照片确认 |
| 买药 | 一般也只能药妆店的要 | 低 | 开放 |

#### B. 宠物基础照顾

| 服务 | 示例 | 风险等级 | 是否开放 |
|---|---|---:|---|
| 遛狗 | 小型犬、短时间散步 | 中 | 谨慎开放 |
| 喂猫 | 按用户说明投喂 | 中 | 开放 |
| 铲屎/清洁 | 猫砂盆、宠物用品简单清洁 | 中 | 开放 |
| 宠物寄养 | 将宠物带回服务人员住处 | 高 | 第一阶段禁止 |

#### C. 临时家务

| 服务 | 示例 | 风险等级 | 是否开放 |
|---|---|---:|---|
| 简单打扫 | 地面、桌面、厨房表面 | 中 | 开放 |
| 垃圾整理 | 分类、打包、搬到指定地点 | 中 | 开放 |
| 收纳整理 | 衣物、杂物、房间整理 | 中 | 开放 |
| 深度清洁 | 空调、油烟机、浴室专业清洗 | 高 | 暂缓 |
| 高处作业 | 窗外、梯子、高空 | 高 | 禁止 |

---

### 5.2 第一阶段禁止服务

以下服务第一阶段不开放：

| 服务 | 禁止原因 |
|---|---|
| 有偿接送乘客 | 可能涉及白タク/道路运输法风险 |
| 医疗护理 | 涉及医疗、护理资格与责任 |
| 老人介护 | 涉及介护保险、专业资格、安全责任 |
| 儿童看护 | 涉及儿童安全、事故责任、许可边界 |
| 宠物寄养 | 涉及动物取扱業登记、事故风险 |
| 按摩/美容 | 可能涉及资格、卫生、安全监管 |
| 开锁/入室代办 | 盗窃、财产、安全风险极高 |
| 大额现金/贵重物品代取 | 纠纷与犯罪风险高 |

## 6. 商业模式设计

### 6.1 收入来源

#### 6.1.1 订单抽佣

平台从每笔订单抽取服务费：

```text
建议初期抽佣比例：10% - 20%
```

示例：

| 订单金额 | 抽佣比例 | 平台收入 | 服务人员收入 |
|---:|---:|---:|---:|
| 3,000日元 | 15% | 450日元 | 2,550日元 |
| 5,000日元 | 15% | 750日元 | 4,250日元 |
| 10,000日元 | 15% | 1,500日元 | 8,500日元 |

#### 6.1.2 紧急服务费

适用于：

```text
30分钟内到达、夜间、雨天、节假日、远距离。
```

#### 6.1.3 会员服务

用户会员：

```text
月费：980日元 - 1,980日元
权益：优先派单、服务折扣、专属客服、免部分手续费。
```

服务人员会员/认证费：

```text
初期不建议收取高额费用，避免供给端增长困难。
可以收取背景审查成本或装备押金。
```

#### 6.1.4 企业合作

可面向：

```text
IT派遣公司、留学中介、房产中介、物业公司、宠物店、语言学校。
```

合作形式：

```text
员工福利、客户增值服务、搬家合作、宠物服务合作、生活支援套餐。
```

## 7. MVP产品功能清单

### 7.1 MVP开发目标

MVP不是做完整平台，而是验证最关键的闭环：

```text
用户发布需求 -> 服务人员接单 -> 平台确认 -> 服务完成 -> 支付/结算 -> 评价/投诉。
```

### 7.2 用户端功能

| 模块 | 功能 | MVP优先级 |
|---|---|---|
| 注册登录 | 手机号/邮箱/Google/LINE登录 | P0 |
| 用户资料 | 姓名、联系方式、默认地址 | P0 |
| 服务分类 | 应急代办、宠物照顾、临时家务 | P0 |
| 发布订单 | 类型、说明、地址、时间、预算、照片 | P0 |
| 订单状态 | 待接单、已接单、进行中、已完成、已取消 | P0 |
| 聊天沟通 | 用户与服务人员文字沟通 | P1 |
| 支付 | 初期可人工确认，后续接Stripe/PayPay | P1 |
| 评价 | 星级、标签、文字、图片 | P0 |
| 投诉 | 投诉原因、证据上传、客服介入 | P0 |

### 7.3 服务人员端功能

| 模块 | 功能 | MVP优先级 |
|---|---|---|
| 服务人员申请 | 基本资料、在留卡/身份证、服务范围 | P0 |
| 审核状态 | 待审核、通过、拒绝、冻结 | P0 |
| 接单大厅 | 查看附近可接订单 | P0 |
| 接单确认 | 接单、拒单、取消原因 | P0 |
| 任务管理 | 待开始、进行中、已完成 | P0 |
| 服务打卡 | 到达、开始、完成、离开 | P0 |
| 收入记录 | 订单收入、平台佣金、结算状态 | P1 |
| 评价查看 | 平均分、用户反馈 | P1 |

### 7.4 管理后台功能

| 模块 | 功能 | MVP优先级 |
|---|---|---|
| 用户管理 | 查询、冻结、解封 | P0 |
| 服务人员审核 | 资料审核、认证记录、黑名单 | P0 |
| 订单管理 | 查看、指派、取消、退款处理 | P0 |
| 投诉管理 | 投诉受理、证据查看、处理结果 | P0 |
| 服务分类管理 | 服务类型、价格规则、开关控制 | P1 |
| 财务管理 | 平台收入、服务人员结算、退款 | P1 |
| 风控管理 | 异常订单、低评分、频繁取消 | P1 |

## 9. 核心业务流程

### 9.1 用户下单流程

```text
用户选择服务类型
      ↓
填写服务说明、地址、时间、预算
      ↓
上传照片或补充说明
      ↓
提交订单
      ↓
系统/管理员匹配服务人员
      ↓
服务人员接单
      ↓
用户确认
      ↓
服务开始
      ↓
服务完成
      ↓
用户评价
      ↓
平台结算
```

### 9.2 服务人员接单流程

```text
服务人员通过认证
      ↓
进入接单大厅
      ↓
查看附近订单
      ↓
确认订单内容和风险
      ↓
接单
      ↓
到达地点打卡
      ↓
开始服务
      ↓
完成服务并上传照片
      ↓
用户确认
      ↓
等待结算
```

### 9.3 投诉处理流程

```text
用户/服务人员提交投诉
      ↓
平台冻结订单结算
      ↓
收集聊天记录、照片、定位、录像授权资料
      ↓
客服初步判断责任
      ↓
必要时联系双方补充说明
      ↓
给出处理结果：退款/补偿/扣分/封禁/报警
      ↓
记录进入风控档案
```

## 10. 质量管理体系

### 10.1 服务人员准入

服务人员必须提交：

```text
真实姓名；
联系电话；
住址或活动区域；
在留卡/身份证明；
银行卡或收款账户；
紧急联系人；
可服务类型；
服务时间段；
头像照片；
简单面谈或视频面试记录。
```

### 10.2 分级认证

| 等级 | 可接服务 | 要求 |
|---|---|---|
| L1基础认证 | 跑腿、代购、送物 | 身份认证 + 电话确认 |
| L2上门认证 | 家务、入室服务 | 身份认证 + 面试 + 培训 |
| L3专项认证 | 专车接送、搬运等 | 追加经验审查/保险/技能确认 |
| L4直营员工 | 高信任服务 | 公司雇佣/制服/设备/培训 |

### 10.3 服务过程记录

建议每个订单保留：

```text
订单内容；
服务人员信息；
用户确认记录；
服务开始/结束时间；
定位打卡；
聊天记录；
服务前后照片；
费用明细；
评价与投诉记录。
```

### 10.4 摄像与录像机制

录像不能默认强制，应采用“用户授权 + 明确告知 + 限定目的 + 限定保存期限”的方式。

建议规则：

```text
用户下单时选择是否开启录像服务；
服务人员入室前再次确认；
录像只用于安全保障、纠纷处理、质量审核；
不得用于宣传、训练AI、第三方销售，除非另行取得明确同意；
录像保存期限建议30-90天；
到期自动删除；
用户可申请查看与删除，法律或纠纷处理需要保留的除外。
```

### 10.5 信用分制度

初始信用分：

```text
100分
```

扣分规则示例：

| 行为 | 扣分 |
|---|---:|
| 无故迟到 | -5 |
| 临时取消 | -10 |
| 服务态度投诉成立 | -10 |
| 未按要求完成服务 | -15 |
| 私下交易 | -20 |
| 重大纠纷 | -30 |
| 偷盗、骚扰、暴力 | 永久封禁并报警 |

## 11. 日本运营合规方案

> 

### 11.1 公司形态建议

#### 阶段一：项目验证

适合：

```text
订单量小、人工派单、尚未正式大规模收费。
```

建议：

```text
进行小规模验证；
记录所有收入与支出；
明确平台规则和免责条款；
避免触碰高许可业务。
```

#### 阶段二：

适合：

```text
融资、品牌扩张、投资人进入、全国化运营。
```

---

### 11.2 特定商取引法 / 通信販売

APP内提供收费服务时，需要注意：

```text
服务提供者信息；
价格与追加费用；
支付方式；
取消规则；
退款规则；
服务开始时间；
投诉联系方式；
利用规约与隐私政策。
```

平台必须清楚告诉用户：

```text
谁提供服务；
平台责任范围；
服务人员责任范围；
订单取消条件；
纠纷处理方式。
```

### 11.3 白タク与接送服务风险

第一阶段禁止提供：

```text
有偿接送乘客；
机场接送；
医院接送；
深夜送人回家；
以“陪同”为名实际进行车辆载客收费。
```

原因：

```text
未取得许可或登记而使用自家用车有偿运送旅客，可能被认定为白タク等违法行为。
```

可做替代方案：

```text
帮助用户叫正规出租车；
陪同用户乘坐公共交通；
代用户搬运行李但不载人；
与正规出租车/ハイヤー公司合作。
```

### 11.4 宠物服务合规风险

宠物服务需要特别谨慎。

相对低风险：

```text
用户家中短时间喂食；
猫砂清理；
短时间遛狗；
按用户书面指示完成照顾。
```

高风险或可能需要登记：

```text
宠物寄养；
长期保管；
宠物训练；
以营业形式持续提供宠物シッター；
把宠物带回服务人员家中；
多人、多宠物、固定营业点运营。
```

建议：

```text
正式开放宠物服务前，向所在都道府县或政令市的动物爱护/动物取扱業窗口确认；
必要时办理第一種動物取扱業相关登记；
服务人员进行宠物安全培训；
购买宠物事故相关保险；
禁止服务人员私自更改喂食、用药、遛狗路线。
```

### 11.5 家政与劳动关系边界

家务服务可能涉及三种模式：

| 模式 | 关系 | 风险 |
|---|---|---|
| 直营雇佣 | 公司雇佣员工提供服务 | 成本高，但可控 |
| 业务委托 | 平台委托自由职业者接单 | 需注意フリーランス法和伪装雇佣 |
| 派遣/供给 | 把人员派到客户处受客户指挥 | 高风险，可能需要派遣许可 |

推荐第一阶段采用：

```text
平台统一定义服务内容；
服务人员按平台标准完成结果；
客户不得直接指挥服务人员从事合同外工作；
所有追加服务必须通过平台确认；
平台保留客服与质量管理权。
```

避免出现：

```text
客户直接命令服务人员长期工作；
按小时把人派给客户随意使用；
平台只收介绍费但不管理服务质量；
形式是外包，实际像派遣。
```

### 11.6 フリーランス委托合规

如果服务人员是业务委托/自由职业者，平台需要：

```text
用书面或电子方式明确交易条件；
明确服务内容、报酬、支付时间、支付方式；
不得无故减少报酬；
不得不合理拒绝结算；
不得利用优势地位强制接受不合理条件；
保留订单与结算记录。
```

建议每单自动生成：

```text
订单委托条件通知书
```

内容包括：

```text
服务内容；
服务地点；
服务时间；
报酬金额；
平台佣金；
支付予定日；
取消规则；
禁止事项；
投诉处理方式。
```

### 11.7 个人信息与录像数据合规

平台将处理：

```text
姓名、电话、地址、在留卡、身份证件、银行卡、定位、聊天记录、照片、录像。
```

因此必须准备：

```text
隐私政策；
个人信息利用目的；
数据保存期限；
第三方提供规则；
用户删除请求流程；
安全管理措施；
内部访问权限管理；
泄露事故应对流程。
```

录像特别注意：

```text
能识别个人的影像通常属于个人信息；
不得无目的长期保存；
不得未经同意用于AI训练或宣传；
不得让无关人员随意查看；
要设置访问日志。
```

### 11.8 保险建议

建议购买或研究：

```text
业务责任保险；
个人赔偿责任保险；
财产损害保险；
宠物事故相关保险；
服务人员意外保险；
网络安全/个人信息泄露保险。
```

## 17. 风险清单与应对策略

| 风险 | 等级 | 应对策略 |
|---|---:|---|
| 上门安全风险 | 高 | 实名认证、录像授权、保险、黑名单、客服介入 |
| 服务质量不稳定 | 高 | 培训、标准流程、评价、分级认证 |
| 法律许可边界不清 | 高 | 先做低风险服务，正式扩张前咨询专家 |
| 白タク风险 | 高 | 禁止有偿载客，只做物品代办或正规交通协助 |
| 宠物事故 | 高 | 限定服务范围、保险、宠物信息表、禁止私自操作 |
| 个人信息泄露 | 高 | 权限控制、加密存储、保存期限、访问日志 |
| 私下交易 | 中 | 平台担保、违约处罚、用户保障权益 |
| 服务人员供给不足 | 中 | 提高报酬、社群招募、服务等级制度 |
| 获客成本过高 | 中 | 华人社群、内容运营、转介绍 |
| 现金流不足 | 中 | 先人工验证，减少早期开发和固定成本 |

## 23. 结论

LifeHero 的核心机会不在于“做一个APP”，而在于解决日本生活服务中最难的三个问题：

```text
信任
调度
标准化
```

第一阶段应该避免重资产和高许可风险，从东京华人市场、低风险服务、人工派单、小区域运营开始。

当完成100单、300单、1000单三个验证节点后，再逐步进入系统化、公司化、合规化、规模化。

最终，这个项目可以从一个“上门服务APP”成长为：

```text
日本华人生活服务平台
      ↓
日本本地即时生活服务平台
      ↓
AI生活服务调度平台
      ↓
家庭机器人服务生态入口
```

真正值得投入的不是一开始做多大，而是从第一单开始，把服务质量、用户信任和运营流程做扎实。
