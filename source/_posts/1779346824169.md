# 1. 项目目标

构建一个：

> 「从单体测试式样书 → YAML中间层 → 自动生成多语言测试代码」的 AI 自动化测试平台

实现：

- 自动生成测试Case
- 自动生成Mock代码
- 自动生成Assert
- 自动生成测试数据
- 自动生成异常测试
- 自动生成边界值测试
- 自动生成多语言测试代码

最终达到：

```text
测试式样书
   ↓
结构化转换
   ↓
YAML
   ↓
代码生成器
   ↓
JUnit / pytest / Jest / NUnit ...
```

# 2. 推荐的整体架构（核心建议）

建议你不要：

```text
式样书 → 直接生成代码
```

而是：

```text
式样书
 ↓
AI解析器
 ↓
统一中间模型（YAML）
 ↓
语言无关测试模型
 ↓
模板引擎
 ↓
多语言测试代码
```

这是非常关键的设计思想。

# 3. 为什么必须加入 YAML 中间层

这是整个系统最核心的设计。

因为：

## 直接生成代码的问题

如果：

```text
式样书 → JUnit
式样书 → pytest
式样书 → Jest
```

你会发现：

- 每种语言都需要重新解析
- 无法复用
- AI提示词无法统一
- 后期维护爆炸

---

## YAML中间层的优势

YAML 本质上是：

> “统一测试语义模型”

例如：

```yaml
testCaseId: TC001

title: 正常登录

input:
  userId: admin
  password: 123456

mock:
  userRepository.find:
    return:
      id: 1
      name: admin

expected:
  result: SUCCESS

assert:
  - result == SUCCESS
```

之后：

```text
YAML → JUnit
YAML → pytest
YAML → Jest
YAML → Go test
```

全部都能生成。

这才是真正的平台化。

# 4. 推荐你增加的“统一测试领域模型”

建议不要直接使用自由YAML。

应该定义：

```text
UTDSL（Unit Test DSL）
```

即：

> 测试领域专用语言

例如：

```yaml
meta:
  id: TC001
  name: 正常登录

target:
  class: LoginService
  method: login

given:
  userId: admin
  password: 123456

mock:
  - target: userRepository.find
    return:
      id: 1

when:
  execute: login()

then:
  assert:
    - result == SUCCESS
```

# 11. 推荐增加“插件化架构”

不要写死。

建议：

```text
LanguageGenerator SPI
```

例如：

```text
generators/
  junit/
  pytest/
  jest/
```

每个语言：

```text
实现统一接口
```

例如：

```java
interface TestCodeGenerator {
    String generate(TestCase testCase);
}
```

这样：

以后新增：

```text
Go
Rust
Kotlin
```

几乎零成本。
